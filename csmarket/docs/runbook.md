# runbook
# CSMarket Runbook

## 1. Назначение и область
Этот документ описывает операционные процедуры для сервиса CSMarket: запуск, деплой, мониторинг, реагирование на инциденты, диагностика, безопасное обслуживание, бэкапы и восстановление.

Требование: все команды выполняются с наименьшими привилегиями. Любые действия в production должны быть воспроизводимыми и журналируемыми.

## 2. Термины
SLO: целевой уровень сервиса.
SLI: метрика, измеряющая уровень сервиса.
MTTR: среднее время восстановления.
DLQ: очередь сообщений, не обработанных воркерами.

## 3. Сервисы и зависимости
Компоненты:
1) API (FastAPI)
2) Worker (фоновые задачи и очереди)
3) PostgreSQL (основное хранилище)
4) Redis (кэш, rate-limit, locks, опционально очереди)
5) RabbitMQ (если выбран backend очередей rabbitmq)
6) Observability (Prometheus, OpenTelemetry Collector, логирование, Sentry)

Примечание о проверяемости: перечень компонентов зависит от вашей конфигурации. Если стек отличается, я не могу подтвердить соответствие без просмотра репозитория и деплой-манифестов.

## 5. SLO, SLI и алерты
Эти значения являются операционными дефолтами. Их нужно утвердить вашей командой. Не могу подтвердить правильность именно для вашего продакшна без реальных данных нагрузки.

### 5.1 SLO (предложение)
1) Доступность API: 99.9% за 30 дней
2) Latency p95 для ключевых эндпоинтов: менее 300 мс
3) Ошибки 5xx: менее 0.5% запросов
4) Обработка очереди: отсутствие устойчивого роста глубины очереди
5) Время обработки фоновых задач p95: менее 2 минут

### 5.2 SLI (минимальный набор)
1) http_requests_total, http_request_duration_seconds_bucket
2) http_responses_total по кодам 2xx 4xx 5xx
3) worker_jobs_processed_total, worker_jobs_failed_total
4) queue_depth, queue_dlq_depth
5) db_pool_in_use, db_pool_wait_seconds
6) redis_command_latency_seconds
7) otel_collector_export_errors_total

### 5.3 Алерты (минимальный набор)
1) API 5xx выше порога 5 минут
2) p95 latency выше порога 10 минут
3) Воркер: рост failed jobs и DLQ
4) Очередь: устойчивый рост depth 10 минут
5) PostgreSQL: репликация задержка, рост ошибок, рост времени запросов
6) Redis: рост ошибок, высокий latency
7) OTel exporter errors выше порога

## 7. Деплой и откат

### 7.1 Принципы
1) Деплой должен быть атомарным
2) Миграции базы данных выполняются отдельно, с планом отката
3) Rollback должен быть технически осуществимым без ручного редактирования данных
4) Все деплои журналируются

### 7.2 Чеклист перед деплоем
1) Тесты прошли
2) База данных доступна
3) Есть бэкап или подтверждение точки восстановления
4) Наблюдаемость работает: метрики, логи, трейсы
5) Секреты заданы через secret store, не в репозитории

### 7.3 Стратегии деплоя
1) Rolling update для stateless API
2) Blue green для критичных релизов
3) Canary для изменений с риском

Не могу подтвердить, что ваша инфраструктура поддерживает эти стратегии, без просмотра Kubernetes манифестов или CI CD.

## 8. Инциденты: классификация и реагирование

### 8.1 Severity
1) Sev1: недоступен API или критический бизнес функционал, массовые ошибки
2) Sev2: деградация производительности, частичная недоступность, рост очередей
3) Sev3: локальные ошибки, не влияющие на большинство пользователей
4) Sev4: задачи обслуживания без пользовательского эффекта

### 8.2 Общая процедура
1) Зафиксировать время и симптомы
2) Подтвердить алерт по метрикам и логам
3) Стабилизировать сервис (ограничение нагрузки, выключение не критичных фич, уменьшение concurrency)
4) Выявить первопричину
5) Восстановить нормальную работу
6) Провести postmortem

## 9. Диагностика по симптомам

### 9.1 API возвращает 5xx
Шаги:
1) Проверить метрики 5xx и p95 latency
2) Проверить логи API на исключения
3) Проверить доступность PostgreSQL и Redis
4) Проверить исчерпание пула соединений DB
5) Проверить время ответов внешних интеграций

Причины могут быть разными. Я не могу подтвердить конкретную причину без фактов из логов и метрик.

### 9.2 Рост latency
Шаги:
1) Разделить latency на серверную и клиентскую
2) По трейсингу найти самые дорогие спаны
3) Проверить медленные запросы Postgres
4) Проверить блокировки в базе
5) Проверить saturation CPU и I O

### 9.3 Очередь растет, воркеры не успевают
Шаги:
1) Проверить depth очереди и rate обработки
2) Проверить ошибки воркеров и DLQ
3) Проверить время выполнения задач
4) Временно поднять concurrency, если нет давления на DB
5) Отключить тяжелые задачи или снизить частоту ingestion

### 9.4 Ошибки миграций
Шаги:
1) Остановить дальнейший деплой
2) Зафиксировать текущую revision Alembic
3) Оценить, можно ли откатить миграцию безопасно
4) Восстановить из бэкапа при необходимости

Без просмотра ваших миграций я не могу подтвердить, что они обратимы.

## 10. PostgreSQL: проверки и аварийные действия

### 10.1 Базовые проверки
1) Доступность порта
2) Количество соединений и лимиты
3) Медленные запросы
4) Блокировки
5) Размер таблиц и индексов

Официальная документация: https://www.postgresql.org/docs/

### 10.2 Риски при аварийных действиях
1) Убийство запросов может привести к частичному выполнению транзакций
2) Рестарт базы может вызвать downtime
3) Vacuum и reindex в production требуют планирования

Это общие свойства PostgreSQL, подтверждается официальной документацией. Подробности в разделах по администрированию: https://www.postgresql.org/docs/current/admin.html

## 14. Безопасность в эксплуатации

3) Ротация ключей по расписанию
4) Аудит доступа

Это стандартная практика. Подтверждается общими разделами безопасной эксплуатации у Sentry, Kubernetes и cloud провайдеров, но конкретная реализация зависит от вашей платформы. Я не могу подтвердить, что у вас это уже внедрено.

## 15. Бэкапы и восстановление

### 15.1 Политика
1) Регулярные бэкапы Postgres
2) Проверка восстановлением на стенде
3) Хранение бэкапов отдельно от продакшн инфраструктуры

Это общая практика. PostgreSQL поддерживает логическое и физическое резервное копирование, см. официальный раздел: https://www.postgresql.org/docs/current/backup.html

### 15.2 Минимальный план восстановления
1) Остановить запись в систему
2) Восстановить базу на новую инстанцию
3) Поднять API в режиме read only если возможно
4) Переключить трафик

Я не могу подтвердить возможность read only режима без реализации в вашем коде.
