# 0007-idempotency
# ADR 0007: Idempotency

## Статус
Принято

## Дата
2026-02-13

## Контекст

csmarket выполняет операции с риском повторного выполнения при ретраях:
- клиентские ретраи из-за сетевых таймаутов
- повторная доставка сообщений/ивентов
- падения воркеров после частичного выполнения
- повторные запросы пользователя (двойной тап, повтор в UI)

Требование: повтор одного и того же намерения не должен приводить к повторной мутации состояния.
Определение идемпотентности для HTTP опирается на семантику intended effect: эффект от нескольких одинаковых запросов совпадает с эффектом одного. RFC 7231 определяет идемпотентные методы и их смысл. :contentReference[oaicite:4]{index=4}

Практика: безопасные ретраи достигаются через идемпотентные API, ключи идемпотентности и дедупликацию. AWS рассматривает идемпотентность как базовый паттерн для безопасных повторов и надёжности. :contentReference[oaicite:5]{index=5}

## Решение

### 1. Обязательный Idempotency-Key для мутаций

Для всех запросов, изменяющих состояние, обязателен Idempotency-Key:
- HTTP POST (создание ресурсов, операции покупки/продажи, вывод/депозит, обмен, списание комиссий)
- любые кастомные команды, которые меняют состояние

Причина: Stripe описывает идемпотентные запросы как механизм безопасного повтора POST при сетевых ошибках посредством ключей. :contentReference[oaicite:6]{index=6}

Исключения:
- HTTP PUT и DELETE могут быть идемпотентными по семантике метода, но в csmarket это не отменяет необходимость бизнес-идемпотентности для сложных доменных операций. RFC 7231 фиксирует перечень идемпотентных методов и определение intended effect. :contentReference[oaicite:7]{index=7}

### 2. Контракт Idempotency-Key

Требования к ключу:
- уникален на бизнес-намерение, генерируется клиентом или gateway
- передаётся в заголовке Idempotency-Key либо в явном поле запроса, но на HTTP уровне предпочтителен заголовок по практике Stripe. :contentReference[oaicite:8]{index=8}
- длина и формат валидируются на входе (ограничение длины и разрешённых символов задаётся на уровне API)

Поведение:
- при первом получении ключа система исполняет операцию и сохраняет итог
- при повторе с тем же ключом система возвращает тот же итог без повторной мутации

### 3. Хранилище идемпотентности

Вводится idempotency store:
- key
- request_fingerprint (хэш нормализованного payload для защиты от повторного использования ключа с другим содержимым)
- status (in_progress, succeeded, failed)
- response_snapshot (минимально необходимый результат для повторного ответа)
- created_at, expires_at

TTL:
- токены регулярно удаляются или истекают по TTL, чтобы контролировать объём хранилища; AWS прямо рекомендует TTL и очистку старых токенов. :contentReference[oaicite:9]{index=9}

Конкурентность:
- insert-if-absent на key
- если уже есть запись in_progress, повторный запрос получает корректный ответ (вариант: 409 или ожидание/пулинг; конкретная политика задаётся на уровне API-контракта)
- если succeeded, возвращается сохранённый response_snapshot

### 4. Защита на уровне базы данных

Для критических доменных сущностей вводятся уникальные ограничения, обеспечивающие дедупликацию даже при сбоях логики приложения.
PostgreSQL поддерживает INSERT ... ON CONFLICT для обработки конфликтов уникальности (DO NOTHING/DO UPDATE). :contentReference[oaicite:10]{index=10}

Правило:
- для каждой операции, где возможны дубликаты, должен существовать детерминированный unique key (например: external_operation_id, payment_intent_id, trade_order_client_id, transfer_idempotency_key)
- запись создаётся через ON CONFLICT, чтобы повтор не создавал дубликат

### 5. Идемпотентность в асинхронной обработке

Для consumers:
- дедупликация по message_id либо по доменному unique key до выполнения side effects
- запись маркера обработки в persistent store с TTL либо без TTL в зависимости от типа события

Для интеграций:
- все исходящие вызовы к внешним сервисам, поддерживающим ключи идемпотентности, должны использовать ключи; Stripe документирует это как безопасный механизм повторов. :contentReference[oaicite:11]{index=11}

### 6. Наблюдаемость и аудит

Логирование:
- ключ идемпотентности логируется как корреляционный идентификатор
- фиксируются исход: first_run или replay
- метрики: количество replay, доля ошибок, доля in_progress конфликтов

Цель:
- отслеживать аномалии ретраев и потенциальные атаки на повторное использование ключей

## Альтернативы

1. Полагаться только на HTTP-семантику (PUT/DELETE)  
Недостаточно для доменных операций; RFC 7231 описывает идемпотентность как intended effect на уровне метода, но бизнес-операции могут включать внешние side effects. :contentReference[oaicite:12]{index=12}

2. Только уникальные ключи в БД без idempotency store  
Не позволяет возвращать одинаковый ответ повторно и не покрывает внешние side effects; ON CONFLICT решает дубликаты данных, но не обеспечивает повтор ответа и контроль выполнения. :contentReference[oaicite:13]{index=13}

3. Только idempotency store без уникальных ограничений  
Уязвимо к багам на уровне приложения; БД-ограничения остаются последней линией защиты. :contentReference[oaicite:14]{index=14}

## Последствия

Плюсы:
- безопасные ретраи без двойных мутаций
- предсказуемость поведения API при сбоях сети
- снижение финансовых и репутационных рисков от дубликатов
- наблюдаемость повторов

Минусы:
- дополнительное хранилище и политика TTL
- усложнение контрактов API и интеграций
- необходимость нормализации payload для request_fingerprint

## Правила реализации

1. Любая мутация должна явно указать, какой ключ идемпотентности используется.
2. Любая доменная сущность, где дубликат критичен, должна иметь уникальный ключ в БД.
3. Любой consumer должен быть идемпотентным и иметь дедупликацию.
4. TTL и очистка токенов обязательны. :contentReference[oaicite:15]{index=15}
