# CSMarket Security

TL;DR: Этот документ задает промышленную модель угроз, требования к защите и контрольные меры для CSMarket (Telegram Mini App, Telegram Bot, gateway-api, events-worker, сервисы домена). Он опирается на проверяемые стандарты OWASP и официальную документацию Telegram.

## 1. Область применения

Компоненты
- Telegram Mini App (miniapp_frontend)
- Telegram Bot (telegram_bot)
- gateway-api (единая точка входа HTTP и realtime)
- events-worker (доставка событий, ретраи, плановые задания)
- доменные сервисы (market-core, wallet-core, ledger-core, steam-core, pricing-core, risk-core, ai-core)

Активы (что защищаем)
- учетные записи пользователей и их права доступа
- состояния сделок, диспутов, эскроу
- финансовые состояния (депозиты, удержания, комиссии, выводы)
- токены и ключи интеграций (Telegram Bot Token и другие)
- события домена (outbox, очередь, уведомления)
- персональные данные (если они появятся)
- данные журналов и аудита

## 2. Модель угроз

Цели атакующих
- захват аккаунта, подмена идентичности
- фрод и злоупотребления (мультиаккаунты, спам, выманивание средств)
- обход лимитов и комиссий
- подмена статусов сделок и платежей
- атаки на доступность (DDoS, exhaustion)
- утечки данных (логи, БД, объектное хранилище)
- supply chain атаки (зависимости, сборка, контейнеры)

Базовые классы угроз соответствуют рекомендациям OWASP API Security Top 10 и OWASP ASVS. Источник: OWASP API Security Top 10, OWASP ASVS.  
https://owasp.org/www-project-api-security/  
https://owasp.org/www-project-application-security-verification-standard/

## 3. Границы доверия и источники истины

3.1 Клиент (Mini App) не является доверенным
- Любые данные из клиента считаются недоверенными и должны быть проверены на сервере. Это базовый принцип OWASP для API. Источник: OWASP API Security Top 10.  
https://owasp.org/www-project-api-security/

3.2 Telegram initData как вход в идентичность
- Для Telegram Web Apps рекомендуется проверять initData на сервере, прежде чем доверять данным пользователя. Источник: Telegram Web Apps.  
https://core.telegram.org/bots/webapps

3.3 Источник истины по критическим состояниям
- Статусы сделок, эскроу и финансовые проводки определяются только сервером (gateway-api и доменные сервисы).
- Realtime сообщения (WS, SSE) являются производной от серверного состояния, а не входом для принятия решений.

## 4. Аутентификация и сессии

4.1 Аутентификация пользователей
- Основной вход: Telegram Web App initData, проверенный на сервере (gateway-api).
- После успешной проверки initData сервер выпускает короткоживущую сессию (например, httpOnly cookie или token), привязанную к устройству и окружению.

Обоснование необходимости серверной валидации initData: Telegram Web Apps.  
https://core.telegram.org/bots/webapps

4.2 Защита от повторного использования (replay)
- initData принимается только в пределах TTL.
- Сервер хранит nonce или fingerprint для короткого окна, чтобы отсекать повтор.

Если конкретная реализация TTL и nonce не описана в официальной документации Telegram, это требование задается как внутренняя политика безопасности и не является утверждением о поведении Telegram.

4.3 Управление сессией
- Хранить сессии в Redis с TTL.
- Отзывать сессию при подозрительной активности (risk-core сигнал).

## 5. Авторизация и права доступа

5.1 RBAC
Роли
- user
- pro
- support
- admin
- system (внутрисервисные задачи)

Правила
- Админские маршруты доступны только admin и только через отдельный admin канал доступа.
- Support может видеть только минимально необходимую информацию и только для задач модерации и диспутов.

Рекомендация принципа наименьших привилегий является общепринятой практикой безопасности; источник как стандарт верификации: OWASP ASVS.  
https://owasp.org/www-project-application-security-verification-standard/

5.2 ABAC для доменных объектов
- Проверка доступа на уровне объекта: сделка, листинг, кошелек, диспут.
- Пользователь видит только свои сделки и свои финансовые операции.

## 6. Идемпотентность и конкуренция

6.1 Idempotency-Key
- Для критических операций (создание сделки, депозит, вывод, подтверждение этапов эскроу) обязателен Idempotency-Key.
- Идемпотентность хранится в Redis или в Postgres таблице idempotency_keys с TTL.

6.2 Optimistic concurrency
- Для обновления статусов сделки и эскроу использовать version или updated_at и проверку ожидаемого значения.

Это раздел фиксирует инженерные меры против повторных запросов и гонок; это не утверждение о внешних системах.

## 7. Защита API

7.1 Валидация входных данных
- Везде использовать строгие схемы (Pydantic) и запрет неизвестных полей.
- Ограничения размеров: payload, заголовки, вложения.

7.2 Rate limiting и anti abuse
- Пер-пользователь, пер-IP, пер-эндпоинт лимиты.
- Отдельные лимиты на создание листингов, создание сделок, запросы статусов.
- Прогрессивные задержки и блокировки при подозрительном поведении.

Класс угроз "Unrestricted Resource Consumption" описан в OWASP API Security Top 10.  
https://owasp.org/www-project-api-security/

7.3 CORS и Origin policy
- Для Web App разрешать только доверенные origins.
- Для админки отдельный origin и отдельные ключи доступа.

7.4 CSRF
- Если используется cookie-сессия, включить CSRF защиту для state changing запросов.
- Если используется Authorization header с short-lived токенами и strict CORS, CSRF риск снижается, но не отменяется.

## 8. Realtime безопасность (WS, SSE)

8.1 Аутентификация соединений
- WS и SSE создаются только при наличии валидной серверной сессии.
- Не использовать параметры URL как единственный механизм авторизации.

8.2 Каналы и изоляция
- Подписки привязаны к user_id.
- Для событий сделки дополнительная проверка владения deal_id.

8.3 Ограничение сообщений
- Ограничить частоту сообщений по WS.
- На сервере жестко ограничить размер кадров.

## 9. События и outbox

9.1 Outbox pattern
- Доменные изменения и запись outbox события выполняются в одной транзакции Postgres.
- events-worker вычитывает outbox и публикует в брокер.

9.2 Доставка и ретраи
- Повторная доставка должна быть безопасна (consumer idempotency).
- Любой consumer обязан обрабатывать дубликаты событий.

Обоснование для idempotency и надежной обработки событий относится к внутреннему дизайну и не требует внешней верификации.

## 10. Платежи, кошелек и учет (wallet-core, ledger-core)

10.1 Единственный источник истины
- Балансы и комиссии считаются только из ledger.
- wallet-core не изменяет баланс напрямую, а формирует события и проводки через ledger.

10.2 Webhooks
- Все webhook входы проверяются подписью провайдера (если провайдер ее предоставляет).
- Все webhook маршруты должны быть изолированы, rate limited и идемпотентны.

Если провайдеры криптоплатежей будут отличаться по возможностям подписи вебхуков, это будет уточняться по официальной документации конкретного провайдера. Не могу подтвердить это без указания провайдера.

10.3 Выводы средств
- Включить риск-проверки (risk-core): лимиты, задержка, дополнительная верификация, ручной review при срабатывании правил.
- Все операции вывода обязаны иметь полный аудит трейл.

## 11. Интеграция Steam (steam-core)

- Любые данные о предметах должны проходить нормализацию и валидацию.
- Не хранить чувствительные секреты в логах.
- Любые автоматические действия должны быть подтверждаемы событиями и иметь аудит.

Я не могу подтвердить детали конкретных Steam API или торговых механизмов без точного перечня используемых официальных endpoints и условий доступа.

## 12. Данные и приватность

12.1 Минимизация
- Хранить только необходимое.
- В PII полях включать политики retention и удаление по правилам продукта.

12.2 Шифрование
- Transport: только TLS.
- At rest: шифрование дисков на инфраструктуре.
- Секреты: только через секрет-хранилище или переменные окружения в защищенной системе деплоя.

12.3 Логи
- Запрет на логирование токенов, initData, ключей, полных payload платежей.
- Маскирование чувствительных полей.

## 13. Секреты и ключи

- Telegram Bot Token хранить только в секрет-хранилище, не в репозитории.
- Регулярная ротация секретов.
- Принцип: один сервис один набор секретов, минимум доступов.

## 14. Supply chain и сборка

- Pin зависимостей и проверка уязвимостей в CI.
- Подпись контейнеров и контроль источников образов.

OWASP ASVS включает требования к безопасной разработке и верификации.  
https://owasp.org/www-project-application-security-verification-standard/

## 15. Мониторинг, аудит и реагирование

15.1 Audit trail
- Отдельная таблица или поток событий для действий: смена статуса сделки, админские решения, выводы средств, блокировки.

15.2 Observability
- request_id, correlation_id, trace_id сквозные для gateway, worker, bot.
- Алерты на аномалии: рост ошибок, рост ретраев, рост отмен сделок, рост спорных кейсов.

15.3 Инциденты
- Runbook: изоляция ключей, ротация токенов, остановка выводов, деградация функций.

## 16. Security checklist (минимальный)

Auth
- серверная проверка initData
- TTL и анти-replay политика
- короткоживущие сессии

API
- строгая схема входа
- rate limit
- idempotency для критических операций
- объектная авторизация

Payments
- webhook signature где возможно
- ledger как источник истины
- аудит и риск-фильтры на вывод

Ops
- секреты вне репозитория
- запрет чувствительных логов
- CI проверки зависимостей
