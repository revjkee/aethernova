# Reva Studio Docs

## О проекте

Reva Studio — промышленная цифровая платформа для beauty-бизнеса с фокусом на масштабируемость, автоматизацию, безопасность и дальнейшее развитие в SaaS-модель.

Платформа проектируется как единая система для:

- онлайн-записи клиентов
- управления мастерами и расписанием
- каталога услуг и цен
- бонусной и токенизированной лояльности
- уведомлений и напоминаний
- аналитики и отчётности
- административной панели
- Telegram-интерфейса
- дальнейшего подключения партнёров, филиалов и новых направлений услуг

Текущий бизнес-контекст:

- стартовая конфигурация — около 8 мастеров
- основные направления — брови, ресницы, ногти, массаж, визаж
- предусмотрено дальнейшее расширение, включая новые услуги и партнёрские сценарии

## Цели документации

Документация в директории `docs/` нужна для того, чтобы:

- быстро вводить в проект новых разработчиков
- фиксировать архитектурные решения
- уменьшать количество скрытых договорённостей
- ускорять поддержку и развитие системы
- снижать риск ошибок при изменении доменной логики
- обеспечивать воспроизводимость локальной, тестовой и production-среды

## Принципы проекта

### 1. Domain-first
Сначала предметная область, потом фреймворк и инфраструктура.

### 2. Production-ready by default
Каждый модуль проектируется так, чтобы путь до production не требовал переписывания архитектуры.

### 3. Security by design
Безопасность учитывается на уровне моделей, API, доступа, секретов, логирования и CI/CD.

### 4. Observability first
Любой важный процесс должен быть наблюдаем: логи, метрики, трассировки, healthchecks, аудит.

### 5. Backward-safe evolution
Изменения в схеме данных, API и бизнес-логике должны внедряться контролируемо и без разрушения системы.

### 6. Explicit over implicit
Лучше явная структура, интерфейсы, типы и контракты, чем скрытая магия.

## Архитектурное видение

Reva Studio строится как модульная backend-платформа с разделением ответственности по слоям и bounded context.

Базовое направление архитектуры:

- `domain` — сущности, value objects, доменные правила
- `application` — use cases, orchestration, DTO, service layer
- `infrastructure` — БД, кеш, брокеры, внешние интеграции
- `presentation` — API, Telegram handlers, admin/UI adapters
- `shared` — общие примитивы, ошибки, базовые контракты, utils строго без нарушения границ домена

## Ключевые доменные области

### Клиенты
Отвечает за профиль клиента, контактные данные, историю посещений, статусы, предпочтения и ограничения.

### Мастера
Отвечает за профили сотрудников, специализацию, графики, статусы занятости и метрики эффективности.

### Услуги
Отвечает за каталог услуг, длительность, стоимость, категории, совместимость и правила отображения.

### Бронирования
Отвечает за запись, перенос, отмену, статусы визитов, конфликты слотов, политики предоплаты и правила уведомлений.

### Лояльность
Отвечает за бонусные баллы, уровни, начисления, списания, условия акций и будущую токенизацию.

### Платежи
Отвечает за регистрацию платежных событий, сверку, статусы оплаты, возвраты и интеграции с платёжными провайдерами.

### Уведомления
Отвечает за отправку сообщений через Telegram, email, push и внутренние события системы.

### Аналитика
Отвечает за агрегаты, KPI, отчёты, retention, загрузку мастеров, выручку, повторные записи и качество сервиса.

## Предполагаемый стек

Ниже перечислен целевой стек проекта. Конкретная реализация может уточняться по мере развития модулей.

### Backend
- Python 3.12+
- FastAPI
- Pydantic v2
- SQLAlchemy 2.x
- Alembic

### Асинхронные задачи
- Celery или APScheduler для планировщика
- Redis как broker/cache в прикладных сценариях

### Хранилище
- PostgreSQL как основная реляционная БД
- Redis для кеша, throttling, ephemeral-state

### Интерфейсы
- Telegram Bot
- Web Admin / Backoffice API
- в дальнейшем — публичный frontend / customer cabinet

### Качество и контроль
- pytest
- mypy
- ruff
- pre-commit

### Инфраструктура
- Docker
- Docker Compose
- `.env`-based config
- далее — CI/CD, registry, staging, production deployment

## Нефункциональные требования

### Масштабируемость
Система должна поддерживать рост от одной студии до multi-branch / partner-ready платформы без полной переработки ядра.

### Надёжность
Критичные операции должны быть идемпотентны или защищены от повторного исполнения.

### Аудит
Изменения статусов записи, начисления бонусов, отмены и административные действия должны быть трассируемы.

### Безопасность
Должны быть разграничены права доступа между:
- клиентом
- мастером
- администратором
- супер-администратором
- сервисными аккаунтами

### Наблюдаемость
Минимально необходимы:
- структурированные логи
- метрики API и background jobs
- healthchecks
- correlation id / request id
- аудит доменных событий
