# ADR 0005: Booking Consistency Model

## Status

Accepted

## Date

2026-03-22

## Owners

- Architecture
- Backend
- Product Operations

## Context

Reva Studio is a booking-centric platform where client trust depends on correct reservation behavior under concurrency, retries, network failures, background processing, and partial integration outages.

The platform must support:

- concurrent booking attempts for the same staff member and time slot
- rescheduling and cancellation flows
- temporary reservation holds during confirmation and payment steps
- retry-safe API behavior
- background reminders and follow-up jobs
- multi-tenant isolation
- auditable booking lifecycle transitions
- future support for prepayment, waitlists, and external calendar sync

The main business risk is double-booking, stale availability, or divergent state between:

- API responses
- database records
- background jobs
- payment state
- notification state
- availability projections

We need a consistency model that is strict where money and slot ownership are involved, but operationally efficient for side effects such as notifications and analytics.

## Decision

We adopt the following booking consistency model.

### 1. Source of truth

The primary relational database is the single source of truth for:

- booking record
- booking status
- slot ownership
- hold expiration
- staff allocation
- tenant boundary
- service duration snapshot
- price snapshot
- client identity link
- idempotency result binding

No cache, queue, or read model may independently decide final slot ownership.

### 2. Strong consistency for slot ownership

Slot ownership is enforced synchronously inside a single database transaction.

Operations that must be transactionally consistent:

- create booking
- confirm booking
- place temporary hold
- convert hold to confirmed booking
- cancel booking
- reschedule booking
- assign or reassign staff if that changes slot occupancy

These operations must execute with row-level protection and conflict detection at the persistence layer.

### 3. Booking state machine

A booking must move only through explicitly allowed transitions.

Canonical states:

- `pending`
- `held`
- `confirmed`
- `cancelled`
- `completed`
- `no_show`
- `expired`

Optional future states:

- `payment_pending`
- `payment_failed`
- `waitlisted`

Transitions must be validated in domain logic, not only at the API layer.

### 4. Temporal hold model

Before final confirmation, the system may place a short-lived hold on a slot.

Hold rules:

- a hold reserves capacity for a bounded TTL
- expired holds are not considered active occupancy after expiration
- hold expiration must be represented by explicit timestamp data
- cleanup may be asynchronous, but availability checks must treat expired holds as inactive even before cleanup runs

This prevents stale temporary locks from blocking legitimate bookings.

### 5. Idempotent write semantics

All externally initiated booking mutations must support idempotent execution.

Covered operations:

- create booking request
- prepayment initiation
- booking confirmation callback
- cancellation request
- reschedule request

Each write command from public clients or integrations must carry an idempotency key or equivalent deduplication token.

The system must persist the outcome bound to that key for a bounded retention window.

### 6. Database-first conflict prevention

Conflict prevention must not rely only on application memory or cache.

Protection strategy:

- transactional checks against overlapping active bookings
- row-level locking where needed
- unique or exclusion-style invariants where schema design permits
- strict filtering by tenant, staff, date interval, and active booking states

The persistence layer is the final arbiter of conflict acceptance or rejection.

### 7. Eventually consistent side effects

The following concerns are eventually consistent and are not part of the critical booking transaction:

- notification dispatch
- analytics events
- CRM synchronization
- marketing triggers
- read-model refresh
- search indexing
- reminder scheduling
- loyalty accrual scheduling
- webhook delivery

These side effects are triggered after the authoritative booking transaction commits.

### 8. Transactional outbox

Domain events generated by successful booking mutations must be written in the same database transaction as the booking change through a transactional outbox pattern.

Examples:

- `booking.created`
- `booking.confirmed`
- `booking.cancelled`
- `booking.rescheduled`
- `booking.completed`
- `booking.no_show`
- `booking.expired`

A background dispatcher publishes outbox records to message brokers or internal handlers after commit.

This prevents lost side effects when the booking write succeeds but asynchronous publishing fails.

### 9. Availability projection policy

Availability shown to users is a projection derived from authoritative booking data.

Rules:

- read models may be cached for performance
- final booking acceptance must always revalidate against the source of truth
- stale availability is acceptable for browsing
- stale availability is not acceptable for final confirmation

This allows high read performance without weakening slot integrity.

### 10. Tenant isolation

All booking reads and writes must be scoped by tenant identity.

Requirements:

- every booking row carries tenant ownership
- every availability query filters by tenant
- every uniqueness or conflict rule is tenant-aware
- cross-tenant collisions are impossible by design

### 11. Payment consistency boundary

Payment is related to booking but is not allowed to create ambiguous slot ownership.

Rules:

- if prepayment is required, booking may start as `held` or `payment_pending`
- slot ownership during payment must have explicit expiration
- successful payment callback must be idempotent
- failed or timed-out payment must release the slot by expiration or explicit transition
- booking confirmation after payment must still verify state validity

Money state and booking state remain coordinated through explicit transitions, not implicit assumptions.

### 12. Notification consistency boundary

Notifications must never define truth.

Rules:

- notification delivery success does not change booking validity
- notification failure does not roll back confirmed slot ownership
- retries are allowed
- notification payloads must be derived from committed state

### 13. Rescheduling model

Rescheduling is modeled as one atomic domain operation with two logical effects:

- release old slot
- acquire new slot

From a consistency standpoint it must behave as a single transactional mutation.

A reschedule must not leave the system in an intermediate visible state where both slots are free or both slots are occupied incorrectly.

### 14. Cancellation model

Cancellation is a domain transition, not deletion.

Rules:

- booking rows are not hard-deleted during normal business flow
- cancelled records remain auditable
- slot occupancy logic excludes cancelled and expired states
- cancellation reason and actor metadata should be preserved where applicable

### 15. Auditability

Every booking mutation must produce an audit trail containing enough information to reconstruct operational history.

Audit dimensions:

- actor type
- actor id
- tenant id
- booking id
- command type
- previous state
- new state
- timestamp
- correlation id
- idempotency key
- relevant metadata snapshot

### 16. Time correctness

All persisted timestamps must be stored in UTC.

Business presentation may use tenant or business timezone, but conflict checks and expiration logic operate on canonical normalized timestamps.

### 17. Failure handling policy

If side effects fail after booking commit:

- the booking remains authoritative
- the outbox record stays retryable
- operators can replay side effects safely
- no manual database edits should be required for normal recovery

If the booking transaction fails:

- no outbox event is considered committed
- no side effect may assume booking success

## Architecture Implications

### Domain Layer

Must include:

- booking aggregate or equivalent transactional domain model
- explicit status transition rules
- overlap policy service
- hold expiration policy
- reschedule policy
- cancellation policy
- idempotent command contract

### Application Layer

Must coordinate:

- command handlers
- transaction boundaries
- idempotency handling
- outbox writing
- audit emission
- post-commit orchestration triggers

### Infrastructure Layer

Must provide:

- repository methods with locking semantics
- overlap-safe queries
- outbox storage and dispatcher
- background expiration job
- retry-safe webhook and notification workers
- clock abstraction
- correlation and tracing support

## Operational Rules

### Active occupancy states

For overlap detection, active occupancy should include:

- `held` when not expired
- `confirmed`
- optionally `payment_pending` when product policy reserves the slot during payment

It must exclude:

- `cancelled`
- `expired`
- `completed`
- `no_show`

### Expired hold behavior

Expired holds must be treated as non-blocking in conflict queries even if cleanup has not yet physically updated status.

A cleanup worker should later normalize stale rows to `expired` for reporting clarity.

### Versioning

Bookings should use optimistic versioning for safe updates where appropriate, especially in internal backoffice edits and integration callbacks.

Version checks do not replace slot conflict checks.

## Alternatives Considered

### A. Cache-first availability locking

Rejected.

Reason:
cache-only locking creates ambiguity if cache and database diverge or if lock expiration and booking commit become inconsistent.

### B. Fully eventual consistency for booking writes

Rejected.

Reason:
slot ownership is a hard business invariant and cannot tolerate asynchronous reconciliation after conflicting writes are accepted.

### C. Hard delete on cancellation

Rejected.

Reason:
this weakens auditability, dispute handling, analytics quality, and operational debugging.

### D. Side effects inside the booking transaction

Rejected.

Reason:
this would increase latency, reduce fault tolerance, and couple slot integrity to external delivery systems.

## Consequences

### Positive

- prevents double-booking at the source of truth layer
- keeps booking writes deterministic
- supports safe retries from clients and integrations
- separates critical consistency from non-critical side effects
- improves auditability and recovery
- scales read traffic through projections without sacrificing write correctness
- creates a clear path for payments, waitlists, and external integrations

### Negative

- implementation complexity is higher than naive CRUD
- requires careful transaction design
- requires outbox infrastructure
- availability browsing may still show short-lived stale data under cache
- rescheduling logic becomes stricter and more complex

## Non-Goals

This ADR does not define:

- final UI behavior
- exact SQL dialect features
- specific message broker technology
- final payment provider contract
- final calendar sync provider
- loyalty calculation formulas

These concerns are addressed by separate ADRs and implementation documents.

## Implementation Rules

The implementation must follow these rules:

1. Never confirm a slot based only on cached availability.
2. Never emit booking side effects before transaction commit.
3. Never use notification success as proof of booking success.
4. Never hard-delete bookings in normal business flow.
5. Always validate state transitions in domain logic.
6. Always scope booking logic by tenant.
7. Always treat idempotency as mandatory for external mutation entry points.
8. Always recheck overlap at final write time.
9. Always preserve audit trail for status-changing operations.
10. Always use UTC in persistence and internal scheduling logic.

## Example Flow

### Create booking with hold

1. Client requests slot.
2. Application validates input and tenant scope.
3. System opens transaction.
4. System checks for overlapping active bookings.
5. If free, system creates booking in `held` state with `hold_expires_at_utc`.
6. System writes outbox event and audit record in same transaction.
7. Transaction commits.
8. Background workers process notifications and timers.

### Confirm held booking

1. Client or payment callback submits confirmation.
2. System resolves idempotency key.
3. System opens transaction.
4. System loads booking with proper concurrency control.
5. System verifies booking is still eligible and hold is not expired.
6. System transitions booking to `confirmed`.
7. System writes outbox event and audit record.
8. Transaction commits.

### Reschedule booking

1. Authorized actor requests new slot.
2. System opens transaction.
3. System locks current booking.
4. System validates transition and checks new slot overlap.
5. System updates interval atomically.
6. System writes outbox event and audit record.
7. Transaction commits.

## Compliance Notes

This ADR is compatible with:

- audit-oriented operations
- personal data minimization, if payload snapshots are controlled
- incident investigation
- replay-based recovery of asynchronous side effects

Sensitive payloads in audit and outbox records must be minimized and masked according to security policy.

## Monitoring Requirements

At minimum, platform monitoring should track:

- booking conflict rejection rate
- hold expiration count
- outbox lag
- outbox retry count
- duplicate idempotency hit rate
- booking state transition failures
- notification delivery failures
- payment callback retry rate
- booking write latency
- reschedule failure rate

## Rollout Strategy

1. Introduce canonical booking states.
2. Introduce overlap-safe transactional write path.
3. Add idempotency storage for external mutations.
4. Add transactional outbox.
5. Move notifications and integrations to post-commit side effects.
6. Add hold expiration worker.
7. Add observability and audit dashboards.
8. Enforce old code path removal.

## Decision Summary

Reva Studio uses a database-authoritative booking consistency model with strong consistency for slot ownership, explicit state transitions, idempotent mutation handling, and eventual consistency for side effects through a transactional outbox.

This is the default rule for all current and future booking-related modules.
