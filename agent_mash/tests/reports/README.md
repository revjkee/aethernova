# Test Reports

## Назначение

Директория `tests/reports` предназначена для хранения результатов автоматизированных тестов,
выполняемых в рамках проекта agent_mash. Отчёты используются для анализа качества,
отладки дефектов, аудита и интеграции с системами CI.

## Общие принципы

- Отчёты являются производными артефактами и не содержат исходных тестовых данных.
- Форматы отчётов должны быть машинно-читаемыми и воспроизводимыми.
- Каждый отчёт должен однозначно соответствовать запуску тестов.
- Отчёты не должны изменяться вручную.

## Структура

Рекомендуемая структура директории:

tests/reports/
├── README.md
├── unit/
│ ├── summary.json
│ └── details.json
├── integration/
│ ├── summary.json
│ └── details.json
├── e2e/
│ ├── summary.json
│ └── details.json
└── coverage/
└── coverage.json


Фактический набор поддиректорий зависит от набора тестов и конфигурации CI.

## Типы отчётов

### Summary отчёты

Содержат агрегированную информацию о запуске тестов:
- количество выполненных тестов
- количество успешных и проваленных тестов
- общее время выполнения
- идентификатор запуска

Используются для быстрого анализа состояния сборки.

### Detail отчёты

Содержат детальную информацию по каждому тесту:
- имя теста
- статус выполнения
- сообщение об ошибке
- стек вызовов
- входные параметры при падении

Используются для отладки и анализа дефектов.

### Coverage отчёты

Содержат данные о покрытии кода тестами:
- общее покрытие
- покрытие по модулям
- покрытие по файлам

Формат зависит от используемого инструмента покрытия.

## Форматы данных

Рекомендуемые форматы:
- JSON для машинной обработки
- XML допускается только при необходимости интеграции со сторонними системами

Каждый файл отчёта должен быть валидным и соответствовать заявленной схеме.

## Требования к именованию

- Имена файлов должны быть в нижнем регистре
- Использовать подчёркивания вместо пробелов
- Имена должны отражать тип тестов и содержимое отчёта

Пример:
login_auth_summary.json
payment_flow_details.json


## Использование в CI

Директория `tests/reports` должна:
- автоматически очищаться перед запуском тестов
- наполняться в рамках одного тестового прогона
- архивироваться или публиковаться как артефакт CI

Конкретная реализация CI зависит от конфигурации пайплайна и не описывается в данном документе. Не могу подтвердить это для всех окружений.

## Хранение и версияция

- Отчёты не должны коммититься в основной репозиторий
- Допускается хранение отчётов в артефактах CI или внешних системах
- README.md является единственным файлом в директории, подлежащим версионированию

## Ответственность

Ответственность за корректность формируемых отчётов несут:
- разработчики тестов
- владельцы CI конфигураций

Любые изменения форматов отчётов должны сопровождаться обновлением данного README.
