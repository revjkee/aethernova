# agent_mash/tests/test_data/README.md

## 1. Назначение

Эта директория содержит тестовые данные, используемые тестами проекта.
Цель: обеспечить воспроизводимые, детерминированные и проверяемые входы и ожидания для тестов, без зависимости от внешних источников.

Тестовые данные здесь являются артефактом качества:
- воспроизводимым,
- минимальным,
- безопасным по содержанию,
- стабильным по контракту.

## 2. Основные принципы

1) Детерминизм
Один и тот же файл входных данных должен давать одинаковый результат при одинаковой версии кода.

2) Изоляция
Тестовые данные не должны требовать сети, внешних сервисов, локальных секретов или времени.

3) Минимальность
Файлы должны содержать только данные, необходимые для проверки поведения.

4) Читаемость
Форматы и структура должны быть понятны без дополнительной документации.

5) Неприкосновенность
Тестовые данные не редактируются тестами во время выполнения. Любая мутация в рантайме запрещена.

6) Безопасность и приватность
Запрещены реальные персональные данные, реальные токены, реальные ключи, реальные пароли пользователей.
Все значения должны быть синтетическими.

## 3. Структура директорий

Стандарт структуры:

tests/test_data/
  README.md
  valid/
  invalid/
  golden/
  schemas/
  fixtures/
  corpus/

Назначение папок:
- valid: корректные входные данные, которые должны проходить валидацию.
- invalid: некорректные входные данные для отрицательных тестов.
- golden: эталонные выходы для регрессионных тестов (golden master).
- schemas: схемы форматов (jsonschema, proto, openapi fragments), если используются.
- fixtures: дополнительные фиксированные структуры, используемые как моки.
- corpus: большой набор примеров для fuzz-like прогонов или расширенных проверок.

Пример, который должен соблюдаться по вложенности:
valid/auth/input/json/login_valid_01.json

## 4. Именование файлов

Общее правило:
<domain>_<scenario>_<variant>_<nn>.<ext>

Примеры:
- login_valid_01.json
- login_invalid_missing_password_01.json
- token_refresh_valid_02.json
- policy_invalid_signature_01.json

Правила:
- только нижний регистр
- слова через подчеркивание
- nn всегда с ведущим нулем: 01, 02, 03
- расширение соответствует формату: json, yaml, txt, bin (только если необходимо)

## 5. Контракт содержимого JSON

Если файл является JSON входом:
1) JSON должен быть валидным (строго).
2) Должны использоваться явные типы: строки, числа, булевы, массивы, объекты.
3) Время не должно зависеть от текущей даты. Если нужно, время фиксируется явным значением.
4) Идентификаторы должны быть синтетическими.

Если файл является JSON ожиданием:
1) Ожидания должны быть устойчивыми.
2) Поля, не являющиеся стабильными (например, случайные id), не должны присутствовать, либо должны нормализоваться тестом до сравнения.

## 6. Golden tests

Папка golden содержит эталонные результаты.
Правила:
- golden-файлы меняются только сознательно при изменении спецификации поведения.
- любое изменение golden-файла должно сопровождаться объяснимым изменением теста, иначе это считается деградацией контроля регрессий.

## 7. Приватность и безопасность данных

Запрещено:
- реальные email, телефоны, адреса, паспортные данные
- реальные ключи, сид-фразы, приватные ключи, API-токены
- дампы продакшн-логов или продакшн-БД

Разрешено:
- синтетические данные (example.com, test.local, user_001)
- предсказуемые токены-заглушки (например, "token_test_01")
- ключи-заглушки, которые явно помечены как тестовые и не подходят для реальных систем

## 8. Версионирование и изменения

Правило совместимости:
- если меняется формат данных, должны появиться новые файлы с новым вариантом,
  старые остаются для регрессионного покрытия, пока не удалены осознанным решением.

Правило истории:
- не перезаписывать файл, если он используется как долгосрочный эталон.
- вместо этого добавлять новый файл или новый вариант.

## 9. Запреты

Запрещено:
- генерировать тестовые данные во время запуска теста и сохранять их обратно в репозиторий
- изменять тестовые данные на диске во время тестов
- хранить бинарники без обоснования
- использовать нестабильные форматы без фиксации спецификации

## 10. Быстрая проверка качества данных

Перед добавлением нового файла:
1) файл минимален и читаем
2) нет секретов и персональных данных
3) формат валиден
4) имя соответствует стандарту
5) файл лежит в правильной категории: valid, invalid, golden
6) тест использует файл по относительному пути внутри tests/test_data

Конец документа.
