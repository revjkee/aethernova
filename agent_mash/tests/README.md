# agent_mash/tests/README.md

## 1. Назначение

Этот документ задает обязательный стандарт тестирования для модуля agent_mash.
Он описывает правила структуры, написания, запуска и контроля качества тестов.

Важно: если какие-либо команды или инструменты из этого стандарта не подключены в репозитории, это не является фактом о проекте. Это требования, которые должны быть зафиксированы в зависимостях и CI, чтобы стандарт стал исполняемым.

## 2. Цели тестового контура

1) Детерминизм
Тесты должны давать одинаковый результат при одинаковых входных данных.

2) Изоляция
Тесты не должны зависеть от внешнего окружения, сети, времени, порядка запуска или состояния других тестов.

3) Скорость
Быстрые тесты запускаются по умолчанию. Тяжелые тесты отделены и запускаются отдельно.

4) Наблюдаемость
Тестовые прогонки должны формировать читаемые отчеты и артефакты.

5) Безопасность
Тесты не должны утекать секреты, отправлять данные наружу, выполнять опасные операции без явного флага.

## 3. Рекомендуемая тестовая пирамида

Unit tests
Проверяют отдельные функции и компоненты без IO.

Integration tests
Проверяют взаимодействие компонентов, включая БД, брокеры, файловую систему, но в контролируемом окружении.

E2E tests
Проверяют сценарии сверху вниз. Должны быть минимальны по количеству.

## 4. Структура директорий

Рекомендуемая структура внутри tests:

tests/
  README.md
  conftest.py
  unit/
  integration/
  e2e/
  fixtures/
  factories/
  data/
  helpers/
  snapshots/

Правила:
- unit не делает сетевые вызовы и не ходит во внешние сервисы.
- integration может поднимать зависимости, но только через явные фикстуры.
- e2e запускается отдельной командой и не должен быть частью дефолтного прогона.

## 5. Именование и стиль

Файлы:
- test_*.py
- *_test.py

Тесты:
- def test_<поведение>_<условие>_<ожидание>():
Пример: test_create_order_when_valid_input_returns_id

Фикстуры:
- имена существительными: user, db_session, event_bus
- фикстуры должны быть локальными и минимальными по области действия

## 6. Маркеры тестов

Рекомендуемые маркеры:
- unit
- integration
- e2e
- slow
- flaky (запрещено без ссылки на тикет и срока устранения)

Правила:
- slow и e2e не входят в дефолтный прогон.
- flaky допускается только временно и требует устранения.

## 7. Запреты и инварианты

Запрещено:
1) Реальные сетевые вызовы в unit тестах.
2) Зависимость от реального времени без стабилизации (заморозка времени или инъекция часов).
3) Использование реальных секретов и токенов.
4) Параллельные гонки по общим ресурсам без изоляции.
5) Скрытые зависимости от порядка запуска.

Инварианты:
- Каждый тест самодостаточен.
- Каждый тест очищает за собой ресурсы.
- Тесты должны быть повторяемы.

## 8. Данные и фикстуры

1) Тестовые данные
- Малые, явные, читаемые.
- Хранятся в tests/data и используются через хелперы.
- Не использовать прод-дампы без анонимизации и разрешения.

2) Factory подход
- Для доменных сущностей использовать factories.
- Не создавать гигантские фикстуры, которые тащат полпроекта.

3) Стабилизация случайности
- Если используется randomness, фиксировать seed.
- Для генерации данных использовать ограниченный набор значений и предсказуемые генераторы.

## 9. Запуск тестов

Этот раздел задает целевой интерфейс запуска. Конкретная реализация зависит от выбранного инструментария, который должен быть добавлен в зависимости и CI.

Минимальный целевой набор команд:

1) Быстрый прогон (unit):
- python -m pytest -m unit

2) Интеграционные:
- python -m pytest -m integration

3) E2E:
- python -m pytest -m e2e

4) Все тесты кроме тяжелых:
- python -m pytest -m "not slow and not e2e"

5) Параллельный запуск (если включен):
- python -m pytest -n auto

6) Покрытие (если включено):
- python -m pytest --cov=agent_mash --cov-report=term-missing

## 10. Отчеты и артефакты

Цель: чтобы CI всегда отдавал артефакты.

Рекомендуемые артефакты:
- junit.xml
- coverage.xml
- htmlcov/
- logs/ (если тесты пишут структурированные логи)

Правило:
- Любая флапа или падение теста должно сопровождаться достаточным логом для диагностики.

## 11. CI контракт

Целевой контракт CI:
1) Unit тесты запускаются на каждый push и PR.
2) Integration тесты запускаются на PR и на main.
3) E2E запускаются по расписанию или вручную, а также на релизных ветках.
4) Покрытие фиксируется и контролируется порогами.
5) Запрещены flaky тесты без тикета и срока устранения.

## 12. Чеклист для нового теста

Перед добавлением:
1) Тест детерминирован.
2) Нет внешней сети.
3) Нет зависимости от времени без стабилизации.
4) Тест изолирован и чистит ресурсы.
5) Название описывает поведение.
6) Маркер установлен корректно.
7) Тест минимален и не тестирует лишнее.

## 13. Диагностика типовых проблем

1) Тест падает только в CI
- Проверь зависимость от путей, времени, локали, часового пояса, порядка запуска, параллелизма.

2) Флапа
- Это дефект. Либо стабилизируй окружение, либо исправь гонку, либо перепиши тест.

3) Медленный прогон
- Вынеси тяжелые тесты в slow и интеграционные, оптимизируй фикстуры, минимизируй поднятие зависимостей.

## 14. Политика изменений

Любое изменение тестового контура должно:
- сохранять детерминизм,
- не ухудшать скорость базового прогона,
- не снижать наблюдаемость,
- не расширять права тестов без необходимости.
