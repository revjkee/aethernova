# Legacy Migrations Archive

## Назначение

Директория `agent_mash/legacy/scripts_archive/migrations` содержит архивные (устаревшие) миграционные скрипты и
одноразовые процедуры преобразования данных, которые больше не являются частью актуального миграционного контура проекта.

Эти материалы хранятся исключительно для:
- форензики и расследований инцидентов, связанных с данными
- восстановления состояния в изолированных окружениях (sandbox, forensic lab)
- проверки исторических трансформаций при регрессионном анализе

Прямое применение на production окружениях запрещено без формального процесса допуска.

## Статус и ограничения

- Архивные миграции не поддерживаются и не гарантируют совместимость с текущей схемой БД.
- Скрипты могут быть неидемпотентны, неатомарны и не соответствовать текущим требованиям безопасности.
- Любые утверждения о конкретных инструментах выполнения (Alembic, Flyway, Liquibase и т.д.) зависят от проекта и не могут быть подтверждены в рамках данного файла. Не могу подтвердить это.

## Политика безопасности

### Запреты

Запрещено:
- запускать архивные скрипты на production без одобрения владельца данных и владельца платформы
- выполнять миграции от имени привилегированных ролей без необходимости
- изменять архивные скрипты задним числом

### Обязательные условия перед запуском

Перед любым запуском в любом окружении необходимо:
1. Сделать полный бэкап целевой БД и проверить возможность восстановления.
2. Зафиксировать текущую версию схемы и приложения (commit/tag, версия контейнера).
3. Подготовить изолированное окружение для прогона (копия данных или синтетический датасет).
4. Включить максимальный уровень журналирования операций миграции (DDL/DML, тайминги, ошибки).
5. Определить критерии успешности и критерии отката.

## Рекомендированный порядок работы

### 1. Классификация миграции

Каждый скрипт должен быть отнесён к одному из типов:
- schema: изменения схемы (DDL)
- data: преобразование данных (DML)
- backfill: заполнение новых полей/таблиц
- cleanup: очистка/сведение данных
- repair: исправление инцидентных состояний

Если тип неочевиден, скрипт должен рассматриваться как high risk.

### 2. Анализ рисков

Перед запуском обязательно:
- оценить объём затрагиваемых строк и предполагаемое время выполнения
- определить наличие блокировок таблиц и влияние на конкурентные транзакции
- выявить зависимости от внешних сервисов или исторических предпосылок данных

### 3. Прогон в sandbox

Минимально необходимый прогон:
- dry-run (если поддерживается инструментом)
- прогон на копии данных
- фиксация метрик: длительность, нагрузка, количество изменённых строк

### 4. Контроль результата

После выполнения обязательно:
- проверить целостность данных (constraints, FK, уникальности)
- выполнить контрольные запросы на ключевые сущности
- сравнить агрегаты до/после (количество пользователей, платежей и т.д.)

## Именование и метаданные

Рекомендуемый формат имени файла:
- `YYYYMMDD_<domain>_<purpose>_<risk>.py|sql`

Пример:
- `20231007_users_backfill_email_low.sql`
- `20231102_payments_repair_duplicates_high.py`

Каждый скрипт должен иметь в начале комментарий-блок:
- назначение
- предпосылки данных
- ожидаемые изменения
- требования к правам
- план отката
- контакт владельца (роль/команда, не персональные данные)

## Аудит и трассируемость

Для каждого факта запуска должен существовать след:
- кто инициировал запуск (учётная запись/роль)
- где запускали (окружение)
- когда запускали (UTC timestamp)
- какой именно артефакт (hash файла/commit)
- результат (успех/ошибка, затронутые сущности)

Рекомендуется сохранять артефакты запуска в `tests/reports/` или в системе CI как build artifacts.
Конкретное расположение зависит от вашего пайплайна. Не могу подтвердить это.

## Архивирование и изменения

- Архивные скрипты не редактируются. Если требуется правка, создаётся новый файл с новым именем и пометкой `patch`.
- Любое удаление скриптов запрещено, кроме случаев формализованного решения о зачистке истории, и только при наличии резервной копии в отдельном хранилище.
