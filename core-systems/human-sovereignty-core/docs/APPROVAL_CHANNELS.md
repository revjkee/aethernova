# APPROVAL_CHANNELS

## Назначение документа

Этот документ описывает систему каналов подтверждения в составе human-sovereignty-core.
Цель системы — гарантировать, что критические действия выполняются только после
осознанного и проверяемого одобрения человеком.

Система каналов подтверждения рассматривается как единый контур безопасности,
а не как набор независимых интерфейсов.

## Базовые принципы

1. Human in the loop  
   Ни одно критическое действие не должно выполняться без подтверждения человека.

2. Zero trust  
   Ни один канал не считается доверенным по умолчанию, включая локальные интерфейсы.

3. Defense in depth  
   Компрометация одного канала не должна приводить к потере контроля.

4. Explicit intent  
   Подтверждение должно содержать явное и недвусмысленное намерение.

5. Auditability  
   Все подтверждения подлежат неизменяемому аудиту.

## Каналы подтверждения

### CLI

Назначение:
- Локальные операции
- Инженерные и аварийные сценарии
- Минимальная зависимость от внешней инфраструктуры

Свойства:
- Требует локального доступа
- Может быть усилен аппаратными ключами
- Поддерживает оффлайн подтверждение

Риски:
- Компрометация хоста
- Социальная инженерия через shell доступ

Меры защиты:
- Mandatory confirmation phrases
- Time based approval window
- Hardware backed secrets при возможности

### WebUI

Назначение:
- Повседневное управление
- Визуализация последствий действий
- Подтверждение сложных операций

Свойства:
- Богатый контекст
- Поддержка предварительного просмотра изменений
- Сессионная модель

Риски:
- XSS
- CSRF
- Компрометация браузера

Меры защиты:
- Strict CSP
- CSRF tokens
- Out of band verification для критических действий

### Messenger

Назначение:
- Out of band подтверждения
- Эскалации
- Асинхронные сценарии

Свойства:
- Независим от основной среды выполнения
- Удобен для подтверждения в движении
- Поддерживает push модель

Риски:
- Компрометация аккаунта
- Перехват сообщений

Меры защиты:
- Подтверждение только с явным challenge id
- Ограниченный набор допустимых команд
- Тайм ауты и отзыв подтверждений

### Hardware

Назначение:
- Высший уровень доверия
- Подтверждение необратимых действий

Свойства:
- Физическое владение
- Минимальная поверхность атаки
- Криптографическое подтверждение

Примеры:
- FIDO2 ключи
- Secure element
- Air gapped устройства

Риски:
- Потеря устройства
- Физическая кража

Меры защиты:
- PIN или биометрия
- Revocation механизмы
- Ограничение области действия ключа

## Иерархия доверия каналов

По умолчанию применяется следующая иерархия:

1. Hardware
2. CLI с аппаратной привязкой
3. WebUI
4. Messenger

Иерархия может быть изменена политиками, но всегда должна быть явно задекларирована.

## Модель подтверждения

### Single channel approval

Допустимо только для:
- Низкорисковых операций
- Обратимых действий

### Multi channel approval

Обязательно для:
- Необратимых действий
- Изменения политик безопасности
- Операций с суверенными ключами

Пример:
- Инициация через WebUI
- Подтверждение через Hardware или Messenger

## Эскалация

Если подтверждение не получено:
- Операция переводится в состояние pending
- Инициируется эскалация в альтернативный канал
- Применяются тайм ауты и автоматический отказ

## Аудит и трассируемость

Каждое подтверждение должно фиксировать:
- Идентификатор операции
- Канал подтверждения
- Контекст запроса
- Временную метку
- Криптографическое доказательство при наличии

Логи должны быть:
- Неизменяемыми
- Коррелируемыми
- Доступными для независимой проверки

## Запрещенные практики

- Автоматическое подтверждение без человека
- Silent fallback на менее доверенный канал
- Повторное использование подтверждений
- Подтверждение без отображения последствий

## Итог

Каналы подтверждения образуют единую систему защиты человеческого суверенитета.
Без согласованной работы всех каналов система считается небезопасной.
