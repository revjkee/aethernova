# Secure Defaults (Human Sovereignty Core)

## 0. Назначение

Этот документ фиксирует безопасные настройки по умолчанию для всех компонентов Human Sovereignty Core: WebUI, API, CLI, фоновые воркеры, аудит и конфигурация.  
Правила сформулированы так, чтобы их можно было реализовать в коде, протестировать и проверять в CI.

## 1. Термины и область

- Компонент: любой исполняемый модуль (WebUI, API, CLI, worker).
- Чувствительная операция: действие, требующее подтверждения человеком, прав доступа или меняющее состояние (approve, deploy, rotate, key-access, policy-change).
- Default-deny: отсутствие явного разрешения означает запрет.
- Immutable audit: аудит-лог только дописывается; корректировки запрещены.

## 2. Базовые принципы

1. Default-deny на всех границах (эндпоинты, RBAC, политики, маршруты).
2. Минимальные привилегии для процессов, ролей, токенов и ключей.
3. Падение в безопасное состояние при ошибке (fail closed).
4. Явные границы доверия: вход всегда считается недоверенным.
5. Тампер-индикация: любые несостыковки аудита, конфигов, подписей трактуются как инцидент.
6. Детализация ошибок наружу минимальная; детали только в защищённых логах.

## 3. Конфигурация по умолчанию

### 3.1 Запуск
- Режим production обязателен для выполнения чувствительных операций.
- DEBUG запрещён в production.
- В production запрещён запуск от root.
- Запрещён небезопасный автоподхват конфигов из непроверенных путей.

### 3.2 Политики конфигов
- Все security-критичные настройки имеют явные значения (никаких implicit defaults).
- Параметры лимитов включены по умолчанию (body size, rate limiting, timeouts).
- Любая нераспознанная настройка конфигурации вызывает ошибку запуска (strict config).

## 4. WebUI и HTTP безопасность

### 4.1 Транспорт
- Только HTTPS в production.
- HSTS включён.
- Cookie только Secure, HttpOnly, SameSite.

### 4.2 Заголовки
По умолчанию включены:
- Content-Security-Policy (CSP)
- X-Content-Type-Options: nosniff
- X-Frame-Options или CSP frame-ancestors
- Referrer-Policy
- Permissions-Policy

### 4.3 Тело запроса
- Лимит размера тела запроса включён.
- Content-Type проверяется по allowlist.
- Запросы без Content-Type запрещены там, где ожидается тело.
- Чтение тела ограничено не только Content-Length, но и фактическим объёмом.

### 4.4 CORS
- CORS выключен по умолчанию.
- Если включён, то:
  - allow_origins строго ограничен.
  - allow_credentials включается только при необходимости.
  - wildcard запрещён в сочетании с credentials.

### 4.5 XSS/DOM
- Любая вставка HTML должна происходить только через безопасные шаблоны или проверенные санитайзеры.
- Запрещено использование eval, Function, string setTimeout/setInterval.
- Обязателен регрессионный тест на XSS (Playwright).

## 5. Аутентификация и сессии

### 5.1 Сессии
- Сессия короткоживущая по умолчанию.
- Ротация сессии после критичных событий (логин, смена прав, MFA).
- Привязка к контексту: user-agent/частичный fingerprint при включённой политике.
- Защита от replay: jti и серверное отслеживание использованных токенов, где применимо.

### 5.2 MFA
- MFA выключать нельзя для привилегированных ролей.
- Для approve-чейн обязателен второй фактор или challenge-binding CLI-WebUI.

### 5.3 Пароли
- Политика сложности, защита от перебора, блокировки, мониторинг аномалий.
- Хранение только через устойчивые хэши паролей (реализация задаётся модулем identity-access).

## 6. Авторизация и RBAC

- Default-deny для всех действий.
- Разрешения задаются только через явные политики.
- Админские действия всегда требуют отдельного подтверждения (human-in-the-loop).
- Все решения авторизации логируются в audit-chain.

## 7. Challenge/Approval flows

- Challenge одноразовый, короткий TTL.
- Challenge подписан и проверяется строго по bindings (aud, sub, op, sid/did при необходимости).
- Любая невалидность или просрочка приводит к отказу.
- Повторное использование challenge трактуется как инцидент.

## 8. Аудит и тампер-детекция

- Audit-лог только append-only.
- Каждый record имеет prev_hash и hash.
- Любая несостыковка (hash mismatch, prev mismatch, duplicate id, non-monotonic seq) считается tamper.
- TamperAlarm обязан:
  - сформировать отчёт;
  - поднять сигнализацию в sinks;
  - инициировать безопасное поведение (например, блокировка чувствительных операций на уровне политики).

## 9. Секреты и ключи

- Секреты не логируются ни при каких условиях.
- Ключи только через безопасное хранилище (keyvault/secret manager) в production.
- Ротация ключей предусмотрена по умолчанию.
- Любая подпись проверяется в constant-time сравнении.

## 10. Логи и наблюдаемость

- Логи структурированные (JSON) по умолчанию.
- PII и секреты редактируются.
- Для security-событий отдельный канал.
- Тайминги и корреляция (trace_id, request_id) включены по умолчанию.

## 11. Runtime hardening

- Ограничения ресурсов включены по умолчанию (CPU/memory).
- Timeouts обязательны для внешних операций и критичных блоков.
- Защита от небезопасных системных вызовов на уровне контейнера/host политики, где применимо.

## 12. Supply chain и сборка

- Версии зависимостей фиксируются.
- SBOM генерируется в CI.
- Подпись артефактов включена.
- Запуск только подписанных образов/пакетов в production, где поддерживается.

## 13. Тесты безопасности (обязательный минимум)

- XSS regression suite (Playwright)
- Проверка лимитов тела запроса
- Проверка default-deny RBAC
- Проверка tamper_alarm на разрыв цепочки
- Проверка environment_guard на production constraints

## 14. Политика нарушений

- Нарушение любого пункта в production:
  - блокирует чувствительные операции;
  - создаёт audit-событие;
  - поднимает alarm через sinks.

## 15. Изменения документа

Любое изменение secure defaults:
- оформляется как policy-change
- требует approve
- фиксируется в audit-chain
